## Library Choices

### React

Used for building a component-based user interface.

### TypeScript

Provides static type checking and improves maintainability.

### Redux Toolkit

Chosen because the application contains several independent features sharing state, including the media collection, uploads and filters.

### Zod

Used for runtime validation of uploaded files. Validation rules are centralized, type-safe and easy to extend.

### CSS Modules

Provides locally scoped styles without introducing additional runtime dependencies.

### Vite

Provides a fast development environment with minimal configuration.

### Browser Cache API

Used to cache generated thumbnails and avoid unnecessary thumbnail regeneration during repeated uploads.

## Architecture

The project follows the Feature-Sliced Design architecture.

```
src
├── app
├── entities
├── features
├── pages
├── shared
├── widgets
└── mock
```

- **app** – application configuration
- **entities** – business entities and their state
- **features** – user scenarios such as uploading and filtering
- **widgets** – composition of multiple features
- **pages** – application pages
- **shared** – reusable utilities, hooks and UI components
- **mock** – mock API implementation

---

## State Management

Redux Toolkit is used for application state management.

Collections are normalized using `createEntityAdapter`.

The application separates persisted media from upload lifecycle state.

### media

Stores successfully uploaded media items.

### uploadMedia

Stores temporary upload tasks, including:

- preparation
- upload progress
- upload errors
- retry state

This keeps upload-specific state isolated from the media collection while allowing files to appear immediately in the gallery during upload.

Objects such as `File` instances and `AbortController`s are intentionally stored outside Redux to preserve store serializability.

---

## Upload Process

Each selected file is processed independently.

The upload pipeline consists of:

1. File validation
2. Thumbnail lookup in the Cache API
3. Thumbnail generation (if needed)
4. File upload
5. Persisting the media item
6. Removing the temporary upload task

Each upload owns its own:

- AbortController
- progress state
- retry logic
- thumbnail generation process

Multiple uploads can run concurrently without affecting each other.

---

## Thumbnail Generation

Images are processed using `createImageBitmap` and the Canvas API.

Video thumbnails are generated by loading video metadata, seeking to the first frame and rendering it to a canvas.

Generated thumbnails are cached using the browser Cache API.

---

## Infinite Scroll

Infinite scrolling is implemented using `IntersectionObserver`.

Additional pages are loaded automatically when the user approaches the end of the gallery.

Duplicate requests are prevented by both the observer conditions and Redux thunk guards.

## Mock API

The project uses a mock API instead of a real backend.

The mock implementation simulates:

- paginated media loading
- upload progress
- upload delays (~500ms)
- upload failures (20% rejects on server, 15% occasional failures while fetching media)

The application communicates with the mock layer through an abstraction rather than accessing mock data directly.

Replacing the mock implementation with a real backend requires changes only inside the API layer.

## Architecture Decisions

### Separate upload state

Upload tasks are stored independently from persisted media items.

This keeps media entities free from temporary upload state and simplifies cleanup after a successful upload.

### Runtime registries

`File` instances and `AbortController`s are stored in runtime registries instead of Redux because they are non-serializable objects.

### Optimistic rendering

Files appear immediately in the gallery as upload cards.

After a successful upload, the temporary upload task is replaced by a persisted media item without reloading the collection.

### Normalized collections

Media entities and upload tasks are stored using `createEntityAdapter`.

This provides efficient updates, fast lookup by id and reusable selectors.

### Thumbnail cache

Generated thumbnails are cached to avoid unnecessary thumbnail generation during repeated uploads or retries.

---

## Trade-offs

- Upload lifecycle is stored separately from persisted media entities. This introduces an additional slice but keeps the media collection free from temporary upload state.

- Runtime-only objects (`File`, `AbortController`) are stored outside Redux to preserve serializability.

- Thumbnail generation is cached using the Cache API. This improves repeated uploads at the cost of using browser storage.

- The application communicates with a mock API through an abstraction layer. Replacing it with a real backend does not require changes in UI components.
